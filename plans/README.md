## Dependency notes

- **001 is the floor.** It makes the local recording the source of truth (IndexedDB),
  so nothing the network does can destroy audio. 002 builds on the same `Session`
  internals and must not regress 001's durability — segment capture is a separate,
  best-effort stream; the continuous storage recording stays the durable, playable artifact.
- **002 depends on backend plan 022** (the `POST audio/segments` endpoint + live transcript
  in `GET :id`). It ships with live transcription **off by default** until backend 022 is
  live, then is flipped on; 404s are harmlessly swallowed when enabled.

## Design decisions settled in review

- Post-resume commit path is **`retry()`**, not `stop()` (a hydrated session is `idle`,
  and `stop()` throws for non-recording status).
- The **`final`-chunk marker is persisted** in IndexedDB meta and restored on hydrate/retry,
  so a resumed recording can still signal completeness.
- The production `IndexedDbChunkStore` is fully implemented and covered by a
  `fake-indexeddb` test — not left as prose.
- Segment capture is fully independent of the durable storage path and cannot regress it.
